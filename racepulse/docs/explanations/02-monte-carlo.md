# 02. 몬테카를로 시뮬레이션

몬테카를로 시뮬레이션은 불확실한 일을 여러 번 가상 실험해서 결과 분포를 보는 방법입니다. 이름은 어렵지만 핵심은 단순합니다.

> 한 번의 예측값만 믿지 말고, 같은 조건에서 10,000번 정도 가상 경주를 돌려서 어떤 결과가 자주 나오는지 보는 방식입니다.

## 왜 RacePulse에 필요한가

경마 예측은 확정 답이 아닙니다. 어떤 말이 1위일 확률이 높아도 실제로는 2위나 3위가 될 수 있습니다.

단순 예측은 보통 이런 식입니다.

```text
1번 말 승률 35%
2번 말 승률 25%
3번 말 승률 20%
4번 말 승률 10%
5번 말 승률 10%
```

하지만 사용자는 더 알고 싶어합니다.

- 이 말이 1위가 아니어도 3위 안에는 자주 들어오는가?
- 인기 없는 말이 이길 가능성은 어느 정도인가?
- 전체 순위 분포가 안정적인가, 불안정한가?

몬테카를로는 이런 질문에 답하기 좋습니다.

## 현재 프로젝트의 구현 위치

```text
ml-server/app/services/monte_carlo.py
backend/src/main/java/com/racepulse/backend/domain/race/service/PredictionService.java
backend/src/main/java/com/racepulse/backend/domain/race/controller/PredictionController.java
frontend/src/pages/race/RacePredictionPage.tsx
```

## 계산 흐름

```text
1. DB에서 해당 경주의 말별 승률 예측값을 읽음
2. 승률 합계가 1이 되도록 정규화
3. 무작위 추첨으로 한 번의 가상 순위를 만듦
4. 이 과정을 10,000번 반복
5. 말마다 1위/2위/3위/4위 이하 비율을 계산
6. 이변 가능성을 계산
7. 결과를 monte_carlo_simulations 테이블에 저장
```

## 확률 정규화란

정규화는 여러 값을 전체 합이 1이 되게 맞추는 과정입니다.

예를 들어 예측값이 다음과 같다고 가정합니다.

```text
A말 30
B말 20
C말 10
```

합계는 60입니다. 정규화하면 다음처럼 됩니다.

```text
A말 30 / 60 = 0.5
B말 20 / 60 = 0.333...
C말 10 / 60 = 0.166...
```

이렇게 해야 무작위 추첨에서 확률로 사용할 수 있습니다.

## `replace=False`가 중요한 이유

현재 구현은 `np.random.choice(..., replace=False, p=probabilities)`를 사용합니다.

`replace=False`는 같은 말이 한 가상 경주 안에서 두 번 나오지 않게 합니다. 경주 순위를 뽑을 때 한 말이 1위와 2위를 동시에 할 수 없으므로 이 설정이 필요합니다.

## 10,000번 반복하는 이유

반복 횟수가 너무 적으면 결과가 흔들립니다.

| 반복 횟수 | 특징 |
| --- | --- |
| 100회 | 빠르지만 결과가 많이 흔들림 |
| 1,000회 | 대략적인 경향 확인 가능 |
| 10,000회 | 속도와 안정성의 균형이 좋음 |
| 100,000회 이상 | 더 안정적이지만 서버 비용 증가 |

RacePulse의 기본값은 10,000회입니다. 운영 중 속도 문제가 생기면 경주 말 수, 요청 빈도, 캐시 정책을 함께 봐야 합니다.

## 이변 가능성 계산

현재 구현에서는 단승 배당이 10.0 이상인 말을 고배당 말로 보고, 그 말이 1위로 뽑힌 비율을 이변 가능성으로 계산합니다.

```text
이변 가능성 = 고배당 말이 1위로 나온 횟수 / 전체 시뮬레이션 횟수 * 100
```

이 기준은 서비스 정책에 따라 바뀔 수 있습니다.

## 결과 저장 방식

시뮬레이션 결과는 `monte_carlo_simulations` 테이블에 저장됩니다.

현재 ML 서버 코드에서는 테이블이 없으면 런타임에 생성합니다. 학습 단계에서는 편하지만, 운영 기준으로는 Flyway 마이그레이션으로 옮기는 편이 더 안전합니다.

운영에서 Flyway가 더 좋은 이유는 다음과 같습니다.

- 어느 환경에서 어떤 테이블이 만들어졌는지 추적 가능
- 배포 전에 스키마 오류를 발견하기 쉬움
- Spring Boot와 FastAPI가 같은 DB 구조를 명확히 공유 가능

## 주의할 점

- 예측값이 없으면 시뮬레이션을 돌릴 수 없습니다.
- 모든 말의 승률이 0이면 정규화가 의미 없어집니다.
- 말 수가 많고 반복 횟수가 높으면 계산 시간이 늘어납니다.
- 무작위 기반이라 매번 완전히 같은 결과가 나오지 않을 수 있습니다.
- 같은 결과를 재사용하려면 저장된 결과의 만료 기준을 정해야 합니다.

## 초보자가 기억할 핵심

몬테카를로는 미래를 정확히 맞히는 마법이 아닙니다. 이미 있는 예측값을 바탕으로 가능한 결과가 얼마나 다양하게 나오는지 보여주는 보조 도구입니다.
