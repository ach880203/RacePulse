# 06. 인증, 카카오 OAuth, JWT, Web Push

이 문서는 RacePulse의 로그인과 알림 구조를 설명합니다.

## 인증이 필요한 이유

모든 사용자가 같은 데이터를 보는 기능은 로그인 없이도 가능할 수 있습니다. 하지만 다음 기능은 사용자를 구분해야 합니다.

- 마이페이지
- 즐겨찾기
- 사용자 설정
- 알림 설정
- 푸시 구독

## Kakao OAuth란

OAuth는 사용자가 외부 서비스 계정으로 로그인하도록 도와주는 표준 방식입니다.

RacePulse에서 카카오 로그인을 쓰면 사용자는 카카오 계정으로 인증하고, RacePulse는 카카오가 확인해준 사용자 정보를 받아 자체 사용자를 생성하거나 조회합니다.

## 카카오 로그인 흐름

```text
1. 사용자가 카카오 로그인 버튼 클릭
2. 카카오 인증 페이지로 이동
3. 사용자가 카카오에서 로그인/동의
4. 카카오가 인가 코드를 RacePulse로 전달
5. 백엔드가 인가 코드로 카카오 토큰 요청
6. 백엔드가 카카오 사용자 정보 조회
7. RacePulse 사용자 생성 또는 조회
8. RacePulse JWT 발급
```

## JWT란

JWT는 JSON Web Token의 약자입니다. 로그인 후 사용자가 API를 호출할 때 본인을 증명하기 위해 보내는 토큰입니다.

```text
Authorization: Bearer 액세스토큰
```

## Access Token과 Refresh Token

| 토큰 | 용도 | 특징 |
| --- | --- | --- |
| Access Token | API 요청 인증 | 유효시간을 짧게 둠 |
| Refresh Token | Access Token 재발급 | 유효시간을 길게 둠 |

Access Token이 길게 살아 있으면 탈취 시 위험합니다. 그래서 짧게 유지하고, 필요할 때 Refresh Token으로 새로 발급합니다.

## Spring Security의 역할

Spring Security는 API 요청 앞단에서 보안 검사를 합니다.

```text
요청 도착
  ↓
JWT 필터가 토큰 확인
  ↓
인증 정보 생성
  ↓
컨트롤러 실행
```

로그인이 필요한 API는 토큰이 없거나 잘못되면 차단됩니다.

## Web Push란

Web Push는 웹 브라우저로 푸시 알림을 보내는 기능입니다.

앱을 설치하지 않아도 브라우저가 알림을 받을 수 있습니다. 다만 사용자의 알림 권한 허용이 필요합니다.

## VAPID란

VAPID는 Web Push에서 서버 신원을 증명하는 키 체계입니다.

브라우저 푸시 서비스는 아무 서버나 알림을 보내게 두지 않습니다. VAPID 공개키/개인키를 사용해서 "이 서버가 알림 발신자"임을 증명합니다.

## 푸시 구독 흐름

```text
1. 브라우저가 사용자에게 알림 권한 요청
2. 사용자가 허용
3. 브라우저가 PushSubscription 생성
4. 프론트엔드가 구독 정보를 백엔드로 전송
5. 백엔드가 DB에 구독 정보 저장
6. 알림 발송 시 백엔드가 Web Push 라이브러리로 전송
```

## 알림 설정

사용자는 알림 종류별로 켜고 끌 수 있습니다. 예를 들면 다음과 같습니다.

- 경주 시작 알림
- 예측 업데이트 알림
- 즐겨찾기 관련 알림

알림 설정 API는 중복 매핑이 생기지 않도록 한 컨트롤러에서만 담당해야 합니다.

## 보안 체크리스트

1. JWT 비밀키는 코드에 직접 넣지 않습니다.
2. 카카오 클라이언트 시크릿은 `.env`나 서버 환경변수로 관리합니다.
3. Refresh Token은 DB나 Redis에서 폐기 가능해야 합니다.
4. 로그아웃 시 토큰 블랙리스트 또는 Refresh Token 삭제 정책을 둡니다.
5. Web Push 개인키는 외부에 노출하지 않습니다.
6. CORS 허용 출처는 필요한 도메인으로 제한합니다.

## 초보자가 기억할 핵심

카카오 OAuth는 "사용자가 누구인지 확인"하는 과정이고, JWT는 "확인된 사용자가 이후 API를 사용할 때 내미는 출입증"입니다. Web Push는 "사용자 브라우저에 알림을 보내기 위한 별도 구독 시스템"입니다.
